Banking PDF Extractor

Intelligent extraction of structured financial data from unstructured banking documents — powered by Gemini, validated by an LLM-as-Judge, and auto-optimised using OPRO prompt tuning.

🤖 Gemini 2.5 Flash ⚖️ LLM-as-Judge 🔄 OPRO Auto-tune 📡 SSE Streaming ⚡ FastAPI + React
Overview

What does it do?

Banks receive thousands of unstructured PDF documents from customers — credit memos, loan agreements, facility letters, bank guarantees. Manually extracting key fields is slow, error-prone, and expensive. This application automates that extraction with high accuracy.

📄

Multi-format PDF Support

Handles both native text-based PDFs and scanned/image-based documents using Gemini's vision capabilities automatically.

🎯

14 Financial Fields

Extracts loan amount, beneficiary name, account number, bank name, interest rate, collateral, guarantor, and more — all in one pass.

⚖️

Accuracy Validation

A second Gemini instance acts as an independent auditor, scoring each extracted field as correct, incorrect, missing, or partial.

🔄

Self-improving Prompts

If accuracy falls short, the system automatically rewrites its own extraction prompt using judge feedback — running up to 3 iterations.

📡

Real-time Progress

Every processing step streams to the UI in real time via Server-Sent Events — no polling, no page refresh.

📦

Batch Processing

Upload multiple PDFs in one shot. Each document is processed sequentially with its own extraction, judgment, and optimisation cycle.

How It Works

Step-by-step pipeline

Each uploaded PDF flows through a four-stage pipeline. The first three stages always run. Stage 4 runs for every iteration regardless of accuracy — always completing all 3 passes to find the best possible extraction.

1

PDF Parsing

PyMuPDF extracts the full text content from each page. If the document is scanned or has fewer than 80 characters per page (image-based), all pages are converted to high-resolution PNG images instead. This dual-mode approach ensures both native and scanned documents are handled correctly.

PyMuPDF · PIL · Text + Vision fallback
2

Field Extraction — Gemini (Extractor)

The document content (text or images) is sent to Gemini 2.5 Flash with a carefully engineered prompt instructing it to extract 14 structured financial fields. The model returns a strict JSON object with exact values as they appear in the document — no paraphrasing or normalisation.

Gemini 2.5 Flash · Structured JSON output
3

Accuracy Evaluation — Gemini (Judge)

A second call to Gemini acts as an independent senior banking auditor. It reads the original document and the extracted fields simultaneously, scoring each field as correct, incorrect, missing, partial, or acceptable (null because it's genuinely absent). It returns an overall accuracy score (0–100) plus specific improvement suggestions.

LLM-as-Judge · Field-level scoring · Improvement feedback
4

OPRO Prompt Optimisation — Gemini (Optimizer)

After each iteration, the judge's feedback is fed to a third Gemini call that acts as a prompt engineer. It reads the current prompt, the failed fields, and the judge's suggestions, then rewrites the extraction prompt to fix those specific issues. This repeats for 3 iterations, always keeping the best-scoring result. The loop is inspired by OPRO (Optimization by PROmpting) — treating the prompt as a parameter optimised by gradient-like feedback from the judge.

OPRO · Iterative prompt optimisation · 3 iterations · Best-of-N selection
OPRO Auto-tune

How prompt optimisation works

Inspired by "Large Language Models as Optimizers" (Yang et al., 2023) and Andrej Karpathy's framing of LLM-driven self-improvement. The extraction prompt is the "parameter" — the judge's score is the "loss" — the optimizer LLM is the "gradient step".

Roadmap

Next steps

The current version is a strong foundation. Here are the natural next enhancements across product, engineering, and AI dimensions.

🎯

Extraction Quality

  • Add ground-truth upload so judge compares against known correct values
  • Support custom field schemas per document type (credit memo vs. LC vs. guarantee)
  • Table detection for documents with structured tabular data
  • Multi-language document support (Arabic, Chinese, etc.)
🔄

OPRO Optimisation

  • Persist best-performing prompts per document type to a prompt library
  • Parallel iteration — run multiple prompt variants simultaneously
  • Few-shot example injection — automatically add successful extractions as examples
  • Confidence scoring per field alongside accuracy
🏗️

Production Readiness

  • PostgreSQL / Supabase for extraction history and audit trail
  • Background job queue (Celery / ARQ) for large batch processing
  • REST API with API key auth for integration with banking systems
  • Docker Compose for one-command deployment
📊

Analytics & Reporting

  • Extraction accuracy dashboard across all processed documents
  • Field-level accuracy heatmap (which fields fail most often)
  • Exportable results as CSV / Excel / JSON
  • Webhook callbacks when extraction completes
🔌

Integrations

  • Direct integration with core banking systems via REST
  • SharePoint / Google Drive as PDF source
  • Email ingestion — process PDFs received as attachments
  • Slack / Teams notifications on completion
🔒

Security & Compliance

  • Role-based access control (maker / checker / admin)
  • PII masking before sending to external AI APIs
  • Audit log of every extraction with user, timestamp, and changes
  • On-premise deployment option using locally-hosted LLMs
