Banking PDF Extractor

Intelligent extraction of structured financial data from unstructured banking documents — powered by Gemini, validated by an LLM-as-Judge, and auto-optimised using OPRO iterative prompt tuning.

🤖 Gemini Flash ⚖️ LLM-as-Judge 🔄 OPRO Auto-tune 📡 Real-time SSE ⚡ FastAPI + React 🗂️ Batch Pipeline 📊 Excel Export 🧪 Mock Dev Mode
Overview

What does it do?

Banks receive thousands of unstructured PDF documents — credit memos, loan agreements, facility letters, bank guarantees. Manually extracting key fields is slow, error-prone, and expensive. This application automates extraction with self-improving accuracy.

📄

Multi-format PDF Support

Handles both native text-based PDFs and scanned/image-based documents using Gemini's vision capabilities — auto-detected per page.

🎯

14 Financial Fields

Extracts loan amount, beneficiary name, account number, interest rate, collateral, guarantor, and more — all from a single document pass.

⚖️

LLM-as-Judge Validation

A second Gemini instance acts as an independent auditor, scoring each extracted field as correct, incorrect, missing, or partial with reasoning.

🔄

OPRO Self-improving Prompts

Judge feedback drives 3 iterations of prompt rewriting. The optimizer targets specific failing fields — the best-scoring result is kept.

How It Works

Step-by-step pipeline

Each uploaded PDF flows through a four-stage pipeline. The OPRO loop always runs all iterations — never early-stops — to find the absolute best prompt and extraction.

1

PDF Parsing — thread-pooled

PyMuPDF extracts full text from each page. If the document has fewer than 80 characters per page (scanned), pages are converted to PNG images. The parsing runs in a thread-pool executor so the event loop stays free to flush SSE events in parallel.

PyMuPDF · PIL · Text + Vision auto-detect run_in_executor — non-blocking
2

Field Extraction — Gemini (Extractor)

The document (text or images) is sent to Gemini with an extraction prompt. The model returns a strict JSON with 14 financial fields. The call runs in a thread executor — the moment it completes, an SSE event fires to the browser showing field count.

Gemini · Structured JSON · Vision fallback Real-time: "✓ 14 fields returned" event
3

Accuracy Evaluation — Gemini (Judge)

A second Gemini call reads the document and extracted fields, scoring each as correct / incorrect / missing / partial / acceptable. The accuracy % and field-level breakdown stream immediately to the UI with colour-coded pills.

LLM-as-Judge · 0–100 score · Field feedback Live field pills: ✓N correct ✗N wrong ∅N missing
4

OPRO Prompt Optimisation — Gemini (Optimizer)

Judge feedback feeds a third Gemini call that rewrites the extraction prompt targeting specific failures. After 3 iterations the best-scoring result is kept. The final tuned prompt is saved and exportable — paste it into the Pipeline JSON settings for fast single-pass future runs (auto_tune: false).

OPRO · Best-of-3 · Prompt delta shown Final prompt export — ready-to-reuse JSON
5

Batch Pipeline — SQLite + Excel Export NEW

Upload a folder of PDFs through the Pipeline tab. Configure auto-tune or paste a pre-tuned prompt as JSON. Results persist in SQLite across restarts. Export a 4-sheet Excel workbook: Summary, Extractions, Field Accuracy heatmap, and OPRO Trajectory.

SQLite · asyncio.create_task · SSE fan-out openpyxl · 4-sheet export · Dashboard
OPRO Auto-tune

How prompt optimisation works

Inspired by "Large Language Models as Optimizers" (Yang et al., 2023). The extraction prompt is the parameter — the judge's score is the loss — the optimizer LLM is the gradient step. All blocking Gemini calls run in thread executors so each sub-step event reaches the browser the moment it fires.

Architecture
